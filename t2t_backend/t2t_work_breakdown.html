⚡ The One Rule — No Bypass Allowed

Every single twin action — regardless of type — must pass through this exact pipeline. There is no shortcut, no direct twin-to-twin call, no exception.

Human → Personal Twin → SID Service → Policy Engine → Agent Router
→ Target Twin → Policy Re-Validation → Orchestration Layer
→ Execution Engine → Audit Log → Notification Service

🔥 Backend Team — Full Work Breakdown
1
SID Service — Identity & Authentication
Twin identity verification · API key auth · Clearance check · Revocation
START HERE
▼
🗄️
Twin Registration Table
Create the database table that stores all registered twins. Each twin must have: twin_id, org_id, hashed API key, role, clearance level, and status (active / suspended / revoked).
models.py db.py
🔑
API Key Authentication (auth.py)
Every incoming request must carry Authorization: Bearer <API_KEY>. The auth layer verifies the key, resolves the twin identity, org, role, and clearance. Returns a verified twin context object used by all downstream checks.
auth.py
🚫
Revocation + Suspension Check
If a twin's status is revoked or suspended, block immediately — no message should proceed. Log a SID_REVOKED event in the audit ledger. This is the security safety net.
auth.py audit.py
2
Policy Engine — Permission Decision Layer
RBAC · ABAC · Threshold checks · ALLOW / DENY / ESCALATE
CRITICAL
▼
⚖️
RBAC — Role-Based Access Control
Define which roles can send which intent types. Example: only CEO and Product roles can initiate PROPOSE_LAUNCH. A Sales twin cannot send EXECUTE_SECURITY_REVIEW. Build a role-permission mapping table.
policy.py
🏷️
ABAC — Attribute-Based Access Control
Layer contextual rules on top of RBAC. Checks: does sender clearance meet the required clearance? Is the action crossing an org boundary without a contract? Is the risk level HIGH? Each check can independently trigger ESCALATE or DENY.
policy.py
📤
Policy Decision Object
Every policy evaluation must return a structured decision object: { "decision": "ALLOW" | "DENY" | "ESCALATE", "rule_id": "...", "reason": "..." }. This object is attached to the audit log entry for every message.
policy.py
🔁
Gate 2 — Policy Re-Validation at Target Side
Policy is checked twice: once at the sender side (Gate 1) and once again at the target twin side before execution begins (Gate 2). This ensures context hasn't changed between routing and execution.
policy.py
3
Message Envelope + Agent Router
Structured message format · /send · /inbox · Dedup · Thread ordering
CORE MODULE
▼
📦
Message Envelope Schema (Pydantic)
Define the strict JSON structure that all T2T messages must follow. Required fields: message_id (UUID), thread_id, sequence_no, from (org + twin + role + clearance), to (org + twin), intent (type + name + risk_level), payload, idempotency_key.
schemas.py
📨
POST /t2t/send — Send Endpoint
This is the entry point for all twin-to-twin messages. Flow: authenticate sender → validate schema → run Policy Gate 1 → check idempotency → store message in DB with state=ROUTED → deliver to recipient inbox → log audit event.
router.py
📥
GET /t2t/inbox/{twin_id} — Receive Endpoint
Target twin polls this endpoint to fetch pending messages addressed to it. MVP approach: polling every few seconds. Returns all ROUTED messages for that twin_id. Marks them as ACKNOWLEDGED on fetch.
router.py
↩️
POST /t2t/reply — Reply Endpoint
Target twin sends back CONFIRM, DECLINE, COMPLETE, or FAIL via this endpoint. Links back via thread_id and message_id. Triggers orchestration if CONFIRM or EXECUTE is received.
router.py
🔁
Idempotency + Deduplication
Store each idempotency_key in Redis with a TTL. If the same key arrives again, return the original response — do not process or re-execute. This prevents double execution of the same action due to network retries.
store.py
🔢
Thread Ordering (sequence_no)
Within a thread_id, messages must be processed in sequence_no order. If a message with sequence_no=5 arrives before sequence_no=4, hold it until the gap is resolved. Prevents out-of-order execution.
store.py
4
Orchestration Layer — Execution Engine
Intent → workflow plan → execute steps → rollback on failure
EXECUTION
▼
📋
Intent Parser + Workflow Planner
When a CONFIRM or EXECUTE intent is received and policy-approved, convert it into a list of ordered workflow steps. Example: EXECUTE_SPRINT_CREATE → [create_jira_ticket, assign_team_members, set_sprint_deadline, notify_stakeholders].
orchestrator/planner.py
🔌
Adapter Interface + Registry
Build a common adapter interface that any tool integration can implement. Register adapters for: Jira, CRM, Google Docs, Calendar, Feature Flags, Deployment Pipelines. Start with one dummy adapter to prove the pattern works before building real integrations.
orchestrator/adapters/base.py orchestrator/adapters/jira.py
🔒
Step-Level Policy Re-Check
Before executing each individual workflow step, re-run the policy check for that specific action. This is important — the spec requires per-step authorization, not just per-message authorization.
orchestrator/executor.py
↩️
Rollback / Compensation Engine
If step 3 of a 5-step workflow fails, automatically reverse steps 1 and 2. Each step definition must include a compensation action (the undo). Example: if create_jira_ticket fails → delete_created_ticket. Logs a ROLLBACK_EXECUTED event.
orchestrator/compensation.py
🧠
Memory Layer Update
After every successful execution, update the relevant memory stores: User Memory (personal preferences + history), Org Memory (shared knowledge base), Decision Graph (record of what was decided and why). This makes twins context-aware over time.
memory/
5
Audit Log Service — Immutable Event Ledger
Every state transition logged · Replayable · Non-repudiation · Compliance
MUST HAVE
▼
📜
Events Table + log_event() Function
Create an append-only events table in the database. The log_event() function must be called at every pipeline stage. Standard fields: event_id, event_type, org_id, twin_id, message_id, policy_decision_ref, severity, result, reason, timestamp.
audit.py models.py
📂
Event Taxonomy — Standard Event Types
All events must use standardized type names for searchability. Required types: MESSAGE_RECEIVED, MESSAGE_VALIDATED, POLICY_ALLOWED, POLICY_DENIED, ROUTED, ACKNOWLEDGED, EXECUTING, COMPLETED, FAILED, ESCALATED, ROLLBACK_EXECUTED, SLA_BREACHED.
audit.py
6
Notification + Escalation Service
Alerts · Human approval workflow · SLA timer · Escalation task
FINAL LAYER
▼
🔔
Notification Sender
After each terminal state (COMPLETED, FAILED, DENIED), send a structured notification to relevant twins. Notification types: INFORMATIONAL (FYI), ACTION_REQUIRED (needs a response), ESCALATION_REQUIRED (human must act), SLA_BREACH (deadline missed).
notifications.py
🚨
Escalation Task Creator
When the Policy Engine returns ESCALATE, create a human approval task with full context: what action was requested, by which twin, risk level, and reason. Start an SLA countdown timer. If approved → resume the flow. If denied → log ESCALATION_DENIED and terminate.
escalation.py
⏱️
SLA Monitoring
Each escalation task and critical intent has an sla_minutes field. Track whether responses are received within the SLA window. If breached, fire a SLA_BREACHED event and escalate further or auto-deny based on configured policy.
escalation.py

PHASE 2
▼
🔌
WebSocket Push (Real-time Delivery)
Upgrade from inbox polling to WebSocket push. Target twin maintains a persistent connection; messages are delivered instantly upon routing instead of being fetched on a timer.
✍️
Ed25519 Message Signing
Cryptographically sign every message envelope with the sender twin's private key. Verifiable at the router — enables non-repudiation (senders cannot deny having sent a message).
🕸️
Full Identity Graph (Neo4j)
Build the complete relationship graph: BELONGS_TO, HAS_ROLE, HAS_CLEARANCE, TRUSTS (via contract), SCOPED_ACCESS. Enables complex permission queries at enterprise scale.
🌍
Cross-Org Contracts + Redaction Profiles
Allow twins from different organizations to communicate — but only when an active contract exists with explicit scope grants. Apply redaction profiles (CROSS_ORG_SAFE, REGULATED_MINIMAL) to strip sensitive fields before cross-org delivery.
